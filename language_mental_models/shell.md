# Shell —— 数据流（Data Flow）

> **核心驱动力：程序就是文本流的加工管道。**
> Shell 程序员不问"我要调用什么命令"，而问"数据从哪里来，经过哪些加工，流向哪里去"。

---

## 心智模型图解

```
Source（来源）
     │
     ▼
Filter（过滤/加工 1）
     │
     ▼
Filter（过滤/加工 2）
     │
     ▼
Sink（去处）
```

每一个命令都是这条流水线上的一个"加工站"：它从标准输入（stdin）读进文本流，
向标准输出（stdout）写出新的文本流，错误信息单独走标准错误（stderr）。
Shell 脚本的复杂度，几乎都来自"如何把这些加工站正确地串起来"。

---

## 核心驱动力详解

在 Shell 的世界里，一切皆文本，一切皆流：

- **命令是过滤器**：输入一段文本流，输出另一段文本流（`grep`、`sort`、`awk`、`sed` 都是典型例子）。
- **管道是连接器**：`|` 把前一个命令的 stdout 接到后一个命令的 stdin，构成流水线。
- **重定向是改道**：`>`、`<`、`2>` 决定这条"水流"从哪来、到哪去。
- **退出码是信号**：命令执行完不是"返回值"，而是向下游发出一个"成功/失败"的信号（`$?`），驱动 `&&`/`||` 这种基于信号的流程控制。

理解了"数据流"这一个核心，Shell 里大部分语法就变成了"给流水线加装不同的零件"，而不是需要死记的孤立命令。

## 典型代码片段

### 1. 管道链（Pipe Chain）—— 最基础的流水线

```bash
cat access.log | grep "ERROR" | sort | uniq -c | sort -rn | head -5
```

**心智模型解读**：这不是"5 个命令"，而是"一条流水线"：`cat` 产生原始文本流 →
`grep` 过滤出含 ERROR 的行 → `sort` 让同类行相邻 → `uniq -c` 把相邻重复行折叠成计数 →
`sort -rn` 按计数倒序 → `head -5` 只取前 5 条。每一步都只关心"进来的流，出去的流"。

### 2. 输出重定向（`>` / `>>`）—— 改变水流的终点

```bash
echo "start" > run.log        # 覆盖写入，水流的终点换成文件
echo "step1 done" >> run.log  # 追加写入，接到已有水流末端
```

**心智模型解读**：`>` 和 `>>` 并不是"保存文件"这么简单的操作，而是把默认流向
终端的 stdout，重新接到了另一个"水槲"（文件）上。

### 3. 错误流合并（`2>&1`）—— 把两条流汇合成一条

```bash
./build.sh > build.log 2>&1
```

**心智模型解读**：`2>&1` 的意思是"把 fd 2（stderr）指向 fd 1（stdout）当前指向的地方"。
顺序很关键——必须先重定向 stdout 到文件，再让 stderr 跟随 stdout，两条流才能真正汇合到同一个文件里。

### 4. 进程替换（`<()`）—— 把命令的输出伪装成一个文件

```bash
diff <(sort file1.txt) <(sort file2.txt)
```

**心智模型解读**：`diff` 只认文件参数，但我们想比较的是"两条命令的输出流"。
进程替换把每条流伪装成一个可读的文件描述符路径（`/dev/fd/N`），从而让"流"能塞进"只接受文件"的命令里。

### 5. Here-Document（`<<EOF`）—— 把一段静态文本当作输入流

```bash
cat <<EOF > config.yaml
name: demo
env: production
EOF
```

**心智模型解读**：here-doc 本质上是临时构造了一条"从脚本内嵌文本到某命令 stdin"的流，
不需要额外文件，就能把多行文本当作输入流喂给下游命令。

### 6. `xargs` —— 把"流中的文本"转换成"命令的参数"

```bash
find . -name "*.log" -mtime +7 | xargs rm -f
```

**心智模型解读**：多数命令的参数来自命令行，而不是 stdin。`xargs` 是流水线里的一个"适配器"，
负责把上游流出来的每一行文本，转换成下游命令的参数列表，从而让"流思维"和"参数思维"接轨。

### 7. `tee` —— 让一条流分叉成两条

```bash
long_running_task | tee full.log | grep "FATAL"
```

**心智模型解读**：`tee` 不消耗流，而是"复制"流：一份原样写到文件（`full.log`），
另一份继续沿管道往下传给 `grep`。心智模型上，这是流水线里罕见的"支流"节点。

### 8. 命令替换（`$()`）—— 把流"收敛"成一个值

```bash
current_branch=$(git rev-parse --abbrev-ref HEAD)
echo "You are on branch: ${current_branch}"
```

**心智模型解读**：和管道（流→流）不同，命令替换是把一条流"收口"成一个字符串值，
再嵌入到另一条命令的参数里。这是"流"与"值"两种思维的转换点。

### 9. `&&` / `||` —— 基于退出码信号的流程控制

```bash
make build && make test && make deploy || echo "Pipeline failed" >&2
```

**心智模型解读**：Shell 里没有传统意义的 if/else 优先，而是"信号驱动"：
每个命令退出后发出 0（成功）或非 0（失败）的信号，`&&` 只在收到"成功"信号时继续，
`||` 只在收到"失败"信号时兜底。这仍然是"数据流"思维的延伸——只是流动的是信号而不是文本。

### 10. `find | xargs` 组合 —— 流水线式的批量文件处理

```bash
find . -type f -name "*.tmp" -print0 | xargs -0 -I{} mv {} /tmp/trash/
```

**心智模型解读**：`find` 产出一条"文件路径流"，`-print0` 用 `\0` 分隔以应对含空格的文件名，
`xargs -0` 按同样的分隔规则消费这条流，逐个转换成 `mv` 命令。全程都是"流的生产—转换—消费"。

### 11. 函数作为自定义过滤器

```bash
to_upper() {
    tr '[:lower:]' '[:upper:]'
}

cat names.txt | to_upper | sort
```

**心智模型解读**：只要一个函数"从 stdin 读，向 stdout 写"，它就能像内置命令一样被塞进管道里。
这说明"过滤器"在 Shell 里不是命令的专利，而是任何遵守"流协议"的东西都能扮演的角色。

### 12. `trap` —— 把信号也纳入"流"的处理范围

```bash
trap 'echo "cleanup on exit"; rm -f "$tmpfile"' EXIT

tmpfile=$(mktemp)
echo "working with $tmpfile"
```

**心智模型解读**：`trap` 让你把"进程即将结束"这个事件，也当作一种需要"处理"的输入，
体现了 Shell 中"一切外部刺激都可以被当作流来加工"的延伸思维。

## 黄金法则

> **不要想着调用命令，而要想着建立数据流。**

写 Shell 脚本前先问自己：数据从哪来（Source）？要经过哪几步加工（Filter）？最终流向哪（Sink）？
想清楚这三点，再决定用哪些命令、要不要重定向、要不要 `xargs`。

## 常见误区对比

### 误区一：用循环代替管道

```bash
# 错误心智模型：像写高级语言一样逐行手动处理
lines=$(cat file.txt)
for line in $lines; do
    if [[ "$line" == *ERROR* ]]; then
        echo "$line"
    fi
done
```

```bash
# Shell 习惯写法：让专门的过滤器去处理流
grep "ERROR" file.txt
```

**为什么后者更好**：`grep` 本身就是为"逐行过滤文本流"高度优化的工具，用 shell 循环
逐行判断不仅慢，还容易在分词（word splitting）、通配符展开上踩坑。

### 误区二：忽略退出码，把 Shell 当成"顺序执行脚本"

```bash
# 错误心智模型：假设每一步都会成功
cd /deploy/target
rm -rf ./*
cp -r /build/output/* .
```

```bash
# Shell 习惯写法：用信号驱动的流程控制兜底
cd /deploy/target || { echo "cd failed, abort" >&2; exit 1; }
rm -rf ./* && cp -r /build/output/* . || { echo "deploy failed" >&2; exit 1; }
```

**为什么后者更好**：一旦 `cd` 失败而脚本继续往下执行 `rm -rf ./*`，很可能在错误的目录里
删掉不该删的文件。Shell 要求你显式处理"信号流"，而不是假设一切顺利。
