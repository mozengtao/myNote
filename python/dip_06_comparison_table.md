## 如何用这张表做技术选型

```
+---------------------------------------------------------------+
|                     选型速查流程                                |
+---------------------------------------------------------------+
| 只是个一次性脚本/内部工具？                                       |
|   -> 用 ③ Duck Typing，别浪费时间定义接口                        |
|                                                                 |
| 需要给函数/事件钩子传"一小段行为"？                                |
|   -> 用 ⑨ Callback Injection 或 ⑩ Functional DI                |
|                                                                 |
| 有"一族可互相替换的算法"（折扣/重试/路由）？                        |
|   -> 用 ⑧ Strategy Pattern，多是配合 Constructor/Method 注入      |
|                                                                 |
| 依赖在对象生命周期内固定不变（Repository/Gateway）？                |
|   -> 用 ② typing.Protocol + ④ Constructor Injection（默认首选） |
|                                                                 |
| 依赖需要在长驻进程运行期被替换（Logger/Metrics 后端）？             |
|   -> 用 ⑤ Setter Injection                                     |
|                                                                 |
| "选择哪个实现"这件事本身逻辑复杂（按配置/环境/能力判断）？           |
|   -> 用 ⑦ Factory + DIP                                        |
|                                                                 |
| 实现数量未知、需要支持第三方扩展（插件市场/多协议驱动）？             |
|   -> 用 ⑪ Plugin Architecture                                  |
|                                                                 |
| 对象图很深、依赖关系复杂、需要区分单例/瞬态生命周期（大型后端）？    |
|   -> 用 ⑫ IoC Container                                        |
|                                                                 |
| 需要强制约束、必须在实例化阶段就发现"没实现完"？                    |
|   -> 用 ① abc.ABC                                              |
+---------------------------------------------------------------+
```

## 一句话总结每种方式的"性格"

```
+------------------------------------------------------------------+
| ①  ABC          : 严格契约，继承强绑定，实例化即校验                |
| ②  Protocol     : 结构契约，零侵入，最 Pythonic 的接口方式          |
| ③  Duck Typing  : 无契约，靠约定，最快但最脆弱                     |
| ④  Constructor  : 依赖一次给齐，默认首选，对象构造完即完整可用       |
| ⑤  Setter       : 依赖可后补/可替换，适合长驻进程热切换             |
| ⑥  Method       : 依赖按调用变化，宿主对象保持无状态                |
| ⑦  Factory      : 把"选择依赖"这件事单独拎出来管理                 |
| ⑧  Strategy     : 一组平等的可替换算法，运行期可互换                |
| ⑨  Callback     : 把"某个时间点该做什么"降维成一个函数              |
| ⑩  Functional   : 把整个依赖降维成函数/闭包，配合函数式组合          |
| ⑪  Plugin       : 核心系统不知道、也不需要知道有多少种实现           |
| ⑫  IoC Container: 把"组装对象图"这件事声明化、自动化、生命周期化      |
+------------------------------------------------------------------+
```
