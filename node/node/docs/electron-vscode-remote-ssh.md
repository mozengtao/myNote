# Electron 框架架构及 VS Code Remote SSH 工作原理

## 一、Electron 框架概述

Electron 是一个使用 Web 技术（HTML、CSS、JavaScript）构建跨平台桌面应用的开源框架，
由 GitHub 于 2013 年为 Atom 编辑器开发，后被 VS Code、Cursor、Slack、Discord 等广泛采用。

核心理念：**一套 Web 代码，打包为 Windows / macOS / Linux 原生桌面应用。**

## 三、进程模型与 IPC 通信

Electron 采用多进程架构，与 Chrome 浏览器的设计一脉相承：

```
┌────────────────────┐          IPC 通道
│    Main Process    │◄═══════════════════════►  操作系统
│    （主进程，唯一）  │                           · 文件系统
│                    │                           · 网络
│  · 应用生命周期     │                           · 硬件设备
│  · 创建窗口        │
│  · 系统级 API      │
│  · Node.js 完整能力 │
└───────┬────────────┘
        │
        │ IPC (ipcMain ◄──► ipcRenderer)
        │ 异步消息传递，序列化 / 反序列化
        │
   ┌────┴──────────────────────────┐
   │              │                │
   ▼              ▼                ▼
┌────────┐  ┌────────┐      ┌────────┐
│Renderer│  │Renderer│ ...  │Renderer│
│  #1    │  │  #2    │      │  #N    │
│        │  │        │      │        │
│ 独立   │  │ 独立    │      │ 独立   │
│ V8 堆  │  │ V8 堆   │      │ V8 堆  │
│ 独立   │  │ 独立    │      │ 独立   │
│ DOM    │  │ DOM    │      │ DOM    │
└────────┘  └────────┘      └────────┘
 各进程内存隔离，一个渲染进程崩溃不影响其他
```

### 关键设计原则

| 特性 | 说明 |
|------|------|
| 主进程唯一 | 整个应用只有一个主进程，掌控全局 |
| 渲染进程多个 | 每个 BrowserWindow 对应一个渲染进程 |
| 进程隔离 | 各渲染进程内存独立，一个崩溃不影响其他 |
| IPC 通信 | 渲染进程不能直接调用 Node API，必须通过 IPC 请求主进程代为执行 |

### V8 引擎的共享

Chromium 和 Node.js 各自内含 V8 引擎，但在 Electron 中它们**共享同一个 V8 实例**：

```
       普通情况                       Electron 中
   ┌────────────┐               ┌────────────────┐
   │  Chrome    │               │   Electron     │
   │  ┌──────┐  │               │   ┌──────┐     │
   │  │  V8  │  │               │   │  V8  │◄─── 共享同一个 V8 实例
   │  └──────┘  │               │   └──┬───┘     │
   └────────────┘               │      │         │
                                │   ┌──┴───┐     │
   ┌────────────┐               │   │      │     │
   │  Node.js   │               │ Chromium Node.js│
   │  ┌──────┐  │               │   │      │     │
   │  │  V8  │  │               │   │ 事件循环   │ │
   │  └──────┘  │               │   │ 也已集成   │ │
   └────────────┘               │   └──────┘     │
                                └────────────────┘
  两个独立的 V8                   一个 V8 + 集成的事件循环
```

Electron 的核心技术挑战之一是**将 Node.js 的事件循环（libuv）与 Chromium 的消息循环合并**，
使两者在同一进程中和谐共存。

## 四、VS Code / Cursor Remote SSH 工作原理

Cursor（和 VS Code）本身就是一个 Node.js 应用

Electron = Chromium + Node.js，这是 VS Code/Cursor 的运行框架
本地端：Electron 完整运行（有 GUI）
远程端：只需要 Node.js（无需 GUI），因为只运行后端逻辑
远程的 cursor-server 就是一个 Node.js 应用，整个编辑器的后端（语言服务、文件操作、扩展宿主、终端管理）都是 JavaScript/TypeScript 编写的，运行在 Node.js 之上

### 4.1 Client-Server 分离架构

Remote SSH 的核心思想是**将编辑器拆成两半**：

```
┌─────────────────────┐        SSH 隧道          ┌──────────────────────────┐
│   Windows 本地端     │ ◄═════════════════════► │     远程 Linux 服务器      │
│                     │                          │                          │
│  Cursor GUI（前端）  │   JSON-RPC over stdio    │  cursor-server（后端）    │
│  - 窗口渲染          │ ◄──────────────────────► │  - 文件系统操作            │
│  - 用户输入          │                          │  - 语言服务（LSP）         │
│  - 主题 / 布局       │                          │  - 终端进程管理            │
│  - 扩展 UI 部分      │                          │  - 扩展的后端部分          │
│                     │                          │  - Git 操作               │
│  运行环境：          │                          │  - 代码搜索 / 索引         │
│  完整 Electron      │                          │                          │
│  (Chromium+Node.js) │                          │  运行环境：               │
│                     │                          │  仅 Node.js（无需 GUI）   │
└─────────────────────┘                          └──────────────────────────┘
```

- **本地端（Client）**：完整 Electron 应用，负责 UI 渲染和用户交互
- **远程端（Server）**：只需要 Node.js 运行时，执行所有后端逻辑

### 4.2 连接建立的完整流程

```
 本地 Cursor                  SSH                    远程 Server
     │                         │                         │
     │  ① 生成安装脚本          │                         │
     │────────────────────────►│                         │
     │  通过 stdin 管道传送     │  ② 执行安装脚本          │
     │                         │────────────────────────►│
     │                         │                         │  检查 ~/.cursor-server
     │                         │                         │  不存在则下载安装
     │                         │                         │  验证 Node.js
     │                         │                         │
     │                         │  ③ 启动 multiplex-server │
     │                         │────────────────────────►│
     │                         │                         │  监听端口，连接复用
     │                         │                         │
     │                         │  ④ 启动 cursor-server    │
     │                         │────────────────────────►│
     │                         │                         │  --host=127.0.0.1
     │                         │                         │  --port=<random>
     │                         │                         │
     │  ⑤ SSH 端口转发 / SOCKS  │                         │
     │◄════════════════════════╪═════════════════════════│
     │                         │                         │
     │  ⑥ JSON-RPC 双向通信建立 │                         │
     │◄───────────────────────►│◄───────────────────────►│
     │                         │                         │
     │ 正常工作：               │                         │
     │ 用户编辑 ──► RPC 请求 ──►│──► 读写文件 ──► 返回结果  │
     │ 语法提示 ◄── RPC 响应 ◄──│◄── LSP 分析 ◄──────────  │
```

#### 各步骤详解

| 步骤 | 动作 | 说明 |
|------|------|------|
| ① | 生成安装脚本 | 本地生成 `cursor_remote_install_xxx.sh`，包含版本检查和下载逻辑 |
| ② | 远程执行脚本 | `type xxx.sh \| ssh -T ovirt bash` 通过 stdin 管道传输执行 |
| ③ | 启动 multiplex | 连接复用层，多窗口共享同一个 SSH 连接 |
| ④ | 启动 code server | 真正的后端服务进程，监听 `127.0.0.1:<random-port>` |
| ⑤ | SSH 隧道 | 通过 `-D <port>`（SOCKS 代理）或 `-L` 端口转发建立安全通道 |
| ⑥ | JSON-RPC | 双向消息协议，本地 UI 与远程后端持续通信 |

### 4.3 远程目录结构 `~/.cursor-server`

这个目录是远程服务器上"无头 Cursor 后端"的完整安装：

```
~/.cursor-server/
├── bin/
│   ├── linux-x64/
│   │   └── <commit-hash>/              # 与本地 Cursor 版本严格对应
│   │       ├── node                     # 自带的 Node.js 运行时
│   │       ├── bin/
│   │       │   └── cursor-server        # 启动入口脚本
│   │       ├── out/
│   │       │   └── server-main.js       # 服务端主程序（JS）
│   │       ├── node_modules/            # 依赖库
│   │       └── package.json             # 模块配置（声明 ES Module 等）
│   └── multiplex-server/
│       └── <hash>.js                    # 连接复用服务
├── data/                                # 远程端的用户数据
│   ├── Machine/                         # 远程专属设置
│   └── logs/                            # 日志文件
└── extensions/                          # 远程端安装的扩展
```

**为什么需要这个目录？**

远程服务器上必须有一个**真实运行的进程**来完成文件读写、代码分析、终端管理等后端工作。
本地 Cursor 只是一个 UI 壳，无法直接操作远程文件系统。

**为什么自带 Node.js？**

cursor-server 的 JS 代码依赖特定版本的 Node.js 运行时。
自带 Node.js 避免了对远程服务器系统环境的依赖，确保版本兼容。

### 4.4 运行时临时文件

cursor-server 启动后会在 `/run/user/<uid>/` 写入状态文件：

```
/run/user/<uid>/
├── cursor-remote-lock.*            # 安装锁，防止并发安装冲突
├── cursor-remote-code.pid.*        # code server 进程 PID
├── cursor-remote-code.token.*      # 连接认证令牌
├── cursor-remote-code.log.*        # code server 运行日志
├── cursor-remote-multiplex.pid.*   # multiplex server PID
├── cursor-remote-multiplex.token.* # multiplex 认证令牌
└── cursor-remote-multiplex.log.*   # multiplex 运行日志
```

## 六、常见问题与排查

### 6.1 远程 cursor-server 启动失败

**典型错误**：`SyntaxError: Unexpected token 'export'`

**原因**：`server-main.js` 使用 ES Module 语法，但 Node.js 以 CommonJS 模式加载，
说明 `package.json`（含 `"type": "module"` 声明）丢失或损坏。

**解决**：

```bash
# 登录远程服务器
ssh <remote-host>

# 清除损坏的安装
rm -rf ~/.cursor-server

# 清理运行时文件
rm -f /run/user/$(id -u)/cursor-remote-*

# 回到本地 Cursor，重新连接即可自动重装
```

### 6.2 确认远程 cursor-server 运行状态

```bash
# 查看进程
ps aux | grep -E 'cursor-server|multiplex' | grep -v grep

# 查看监听端口
ss -tlnp | grep node

# 查看运行时文件
ls -la /run/user/$(id -u)/cursor-remote-*

# 查看日志
tail -50 /run/user/$(id -u)/cursor-remote-code.log.*
```

### 6.3 反复安装失败的可能原因

| 原因 | 排查方法 |
|------|---------|
| 网络问题 | 检查远程服务器外网连通性，cursor-server 包需从互联网下载 |
| 磁盘空间不足 | `df -h ~` 确认 home 目录剩余空间（需约 500MB） |
| 权限问题 | `ls -la ~/.cursor-server` 确认目录属主 |
| Node.js 版本不兼容 | 检查自带 Node 版本：`~/.cursor-server/bin/linux-x64/*/node --version` |
| 防火墙 / 代理 | 确认 SSH 端口转发(`/etc/ssh/sshd_config: AllowTcpForwarding yes`)未被阻断 |
