# sed 核心区别：基础正则 (BRE) vs 扩展正则 (ERE)

使用 `-E` 选项可以启用 **扩展正则表达式 (ERE)**，其核心价值在于简化语法，减少反斜杠（斜杠）的使用，使正则表达式更易读。

---

## 1. 元字符语法对比表

| 功能 | 基础正则 (默认) | 扩展正则 (`-E`) | 说明 |
| :--- | :--- | :--- | :--- |
| **分组 (Grouping)** | `\( ... \)` | `( ... )` | 用于后向引用 `\1`, `\2` |
| **次数：1次或多次** | `\+` | `+` | 匹配至少一个前导字符 |
| **次数：0次或1次** | `\?` | `?` | 匹配可选的前导字符 |
| **精确次数限定** | `\{n,m\}` | `{n,m}` | 例如 `{3}` 匹配 3 次 |
| **逻辑 "或" (OR)** | `\|` | `\|` | 匹配多个分支中的一个 |

---

## 2. 实际代码对比

### 场景 A：提取括号中的内容 (后向引用)
- **默认 (BRE):** `sed 's/ID:\([0-9]*\)/\1/'`
- **使用 -E (ERE):** `sed -E 's/ID:([0-9]+)/\1/'`

### 场景 B：匹配多个连续空格
- **默认 (BRE):** `sed 's/  */ /g'` (或者使用 `\+`)
- **使用 -E (ERE):** `sed -E 's/ +/ /g'`

### 场景 C：同时匹配多个关键词 (或)
- **默认 (BRE):** 极少数版本支持 `\|`，兼容性差
- **使用 -E (ERE):** `sed -E 's/apple|orange/fruit/g'`

---

## 3. 使用建议

1. **可读性优先**：当正则表达式包含 `()`、`+`、`?` 或 `{}` 时，强烈建议使用 `sed -E`。
2. **跨平台性**：现代 GNU sed (Linux) 和 BSD sed (macOS) 均已支持 `-E` 选项，是事实上的行业标准。
3. **避免“斜杠地狱”**：`-E` 可以有效避免因过度转义导致的 `\(\(\(\)\)\)` 这种难以维护的代码。
