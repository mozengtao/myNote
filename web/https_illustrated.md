# HTTPS 工作过程总结

## 1. TCP 连接建立
- 客户端与服务器通过 **TCP 三次握手** 建立可靠连接：
  1. 客户端发送 `TCP SYN`
  2. 服务器回复 `TCP SYN + ACK`
  3. 客户端发送 `TCP ACK`
- 完成后，客户端与服务器之间的基础通信通道建立。

## 2. TLS 握手：证书验证与协商阶段
- **Client Hello**：客户端向服务器发送支持的 TLS 版本、密码套件、随机数等信息。
- **Server Hello**：服务器从客户端选项中选择 TLS 版本和密码套件，回复给客户端。
- **Certificate 发送**：服务器将包含自身**公钥**的数字证书发送给客户端，客户端验证证书合法性（确认服务器身份）。
- **Server Hello Done**：服务器告知客户端 hello 阶段结束。

## 3. TLS 握手：会话密钥交换阶段
- **Client Key Exchange**：客户端生成**会话密钥（对称密钥）**，使用服务器公钥（非对称加密）加密后发送给服务器。
- **Change Cipher Spec（客户端）**：客户端告知服务器后续通信将使用新协商的加密套件。
- **Finished（客户端）**：客户端发送加密完成消息，确认密钥交换成功。
- **Change Cipher Spec（服务器）**：服务器告知客户端已准备好使用加密通信。
- **Finished（服务器）**：服务器发送加密完成消息，确认密钥交换成功。
- 此时双方均持有相同的会话密钥，且仅通信双方知晓。

## 4. 加密数据传输
- 客户端与服务器使用**会话密钥（对称加密）**对 HTTP 数据进行加密后传输。
- 加密后的数据即使被截获，也无法被第三方解密，保证了数据的机密性与完整性。

---

### 补充说明
- **非对称加密**：仅用于会话密钥的安全交换（如 RSA），因其计算开销大，不适合批量数据传输。
- **对称加密**：用于实际业务数据的加密传输（如 AES），效率更高，适合大量数据交互。
- **TLS 版本差异**：图中流程基于 TLS 1.2（需 2 次网络往返），TLS 1.3 优化了握手流程，将往返次数减少至 1 次，性能更高；同时 TLS 1.3 不再支持 RSA 密钥交换，更推荐使用 Diffie-Hellman 算法。
