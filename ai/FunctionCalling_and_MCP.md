# Function Calling 与 MCP 的协同工作机制

## 概述

Function Calling（函数调用）和 MCP（Model Context Protocol，模型上下文协议）是大语言模型（LLM）与外部世界交互的两个互补且协同的关键技术。它们分别解决不同层面的问题，共同构成"意图识别 → 标准化执行 → 结果反馈"的完整工具调用闭环。

下面从定位差异、协同机制、工作流程、实际案例四个维度详细解析它们如何协同工作。

## 核心定位差异（Why 需要两者？）

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| **角色** | 模型的"决策层" | 工具调用的"标准通信层" |
| **关注点** | 模型是否需要调用工具？调用哪个？参数是什么？ | 如何安全、统一、高效地执行这个调用？ |
| **输出/输入** | 模型输出结构化调用指令 | 接收指令并执行，返回结果给模型 |
| **绑定性** | 通常与特定模型强绑定（如 OpenAI、Claude） | 独立于模型，任何支持 MCP 协议的模型均可使用 |
| **类比** | 外卖订单系统（决定点什么餐） | 外卖配送平台（统一取餐、送餐标准） |

> ✅ **关键结论**：Function Calling 负责"想做什么"，MCP 负责"怎么做"。

## 协同工作机制（How 它们一起工作？）

典型的协同流程如下（以用户问"北京明天天气如何？"为例）：

### 工作流程

1. **用户输入**  
   ```
   "北京明天天气怎么样？"
   ```

2. **大模型生成 Function Call**  
   模型识别需调用 `get_weather`，输出：
   ```json
   {
     "name": "get_weather",
     "arguments": {"city": "北京", "date": "2026-03-20"}
   }
   ```

3. **应用层将 Function Call 转换为 MCP 请求**  
   通过适配器（如 mcp-adapter）包装成标准 MCP JSON-RPC 格式：
   ```json
   {
     "jsonrpc": "2.0",
     "id": 123,
     "method": "tools/call",
     "params": {
       "name": "get_weather",
       "arguments": {"city": "北京", "date": "2026-03-20"}
     }
   }
   ```

4. **MCP Client 发送请求给 MCP Server**  
   MCP Server 可能是本地服务（如 STDIO）或远程 API（如 HTTP + SSE）

5. **MCP Server 执行工具调用**  
   调用真实天气 API，获取结果：`{"temp": 15, "condition": "晴"}`

6. **结果通过 MCP 返回 → 注入模型上下文**  
   模型收到结构化结果，生成自然语言回答：  
   > "北京明天晴，气温15℃，建议穿薄外套。"

## 技术架构中的分层协作

```
[用户]
   ↓
[大模型] ——(Function Calling)——→ [应用层 / Agent]
                                      ↓ (转换为 MCP)
                              [MCP Client] ←→ [MCP Server]
                                      ↓
                          [外部工具：API / DB / 本地程序]
```

### 各层职责

- **Function Calling**：模型原生能力（如 GPT 的 tool_calls 字段）
- **MCP**：独立协议层，提供：
  - 统一工具注册与发现
  - 权限控制 & 审计日志
  - 熔断/重试/限流等企业级能力
  - 跨设备、跨云、本地/远程混合调用

## 典型协同场景示例

### 场景：企业智能客服需同时查 CRM + 发邮件 + 读本地日志

| 步骤 | 技术组件 | 作用 |
|------|----------|------|
| 1. 用户问："张三最近一次订单状态？" | LLM + Function Calling | 模型决定需调用 `query_crm(customer="张三")` |
| 2. 系统将调用转为 MCP 请求 | Adapter | 标准化格式，附带用户权限 token |
| 3. MCP 路由到 CRM 服务 | MCP Server | 鉴权通过后查询云端 CRM |
| 4. 若需通知客户，再调 `send_email()` | LLM 再次触发 FC → MCP | MCP 调用邮件服务 |
| 5. 同时读取本地服务器日志分析异常 | MCP 支持 STDIO 协议 | 直接访问本地文件 |

### 对比分析

💡 **若只用 Function Calling**：需为每个工具写硬编码适配，无法复用、难审计。  
✅ **引入 MCP**：所有工具统一接入，权限集中管控，日志统一收集。

## 未来趋势：融合而非替代

- **厂商推动融合**：
  - OpenAI、Anthropic 等厂商正推动融合
  - 提供 Function Call → MCP 的自动转换网关
  - MCP 工具描述开始借鉴 Function Calling 的 Schema 定义方式

- **最终架构演进为三层**：
  1. **意图层（Function Calling）**：模型决定"做什么"
  2. **协议层（MCP）**：标准化"怎么调用"
  3. **执行层（Tools/Services）**：具体实现功能

## 总结

Function Calling 是大模型的"嘴"（说出要调用什么），MCP 是它的"手"（标准化地去执行）——二者协同，让大模型从"光说不练"变为"言出必行"。

这种分层设计既保留了模型的智能决策能力，又通过标准化协议实现了企业级的安全、可扩展与互操作性，是构建下一代 AI Agent 系统的核心范式。
