# AWK "锚点定位 + 相对偏移取值" 心法

## 1. 为什么需要这套心智模型

命令行工具、日志、报表类文本的字段**列数或位置常常不固定**：

- 续行的前导列被省略为空，awk按空白分割时空字段直接消失，导致后面所有字段整体左移
- 不同工具/不同版本输出的列顺序不一致，甚至列数不同
- 一条逻辑记录被拆成了好几行文本
- 一行里数值字段的个数本身是可变的（不定长列表）

在这些场景下，写死 `$3`、`$5` 这类固定下标，只在“样例恰好对齐”的那部分行有效，稍微换一种输入就会取错值。

## 2. 核心思想：两步走

1. **锚点（Anchor）**：在记录（一行/一个字段/一段文本）里找一个具有*独特、可以用正则或字符串唯一识别*的标志物。它的绝对位置可能变化，但“是不是它”很容易判断。
2. **偏移（Offset）**：锚点定位好之后，用**结构上稳定不变的相对关系**（字段偏移量、行偏移量，或“扫描到终止符为止”的动态区间，或按名字查表）去取真正需要的值。

一句话概括：**位置会变，结构不会变** —— 用结构上稳定的相对关系替代绝对下标。

## 3. 通用实施步骤（checklist）

1. 拿到样例数据，标出哪些列/行的位置会变化，哪些“相邻关系”始终不变
2. 选锚点：找一个独特的正则/关键字/文本模式，确认它在所有样例里都不会误命中别的字段
3. 定位锚点：遍历字段（`for i=1..NF`）或遍历行（`NR`/状态标志），用正则匹配找到锚点所在位置
4. 应用偏移，常见有四种手法：
   - **固定字段偏移**：`$(i+k)`，锚点和目标字段之间永远隔着固定数量的列
   - **动态区间扫描**：从锚点开始循环，直到遇到某个终止条件为止，适合不定长的数值列表
   - **表映射（按名字查表）**：先扫表头把列名记录成下标，之后统一按名字取值，天然兼容列序变化
   - **行方向偏移**：锚点是一整行，真正的值在下一行（或下N行），用状态变量把“找到锚点”这件事带到下一条记录
5. 处理边界：锚点没命中怎么办（跳过/清零）、一行里有多个候选怎么选（`break`/取第一个）
6. 用变量或关联数组收集结果，在 `END{}` 里统一输出或汇总

## 4. 六个典型示例

前5个例子分别对应上面4种偏移手法（外加1个“偏移为0”的退化情形），是为讲解思路构造的典型输入；第6个换成本仓库真实生产代码作为印证。每个示例都给出**典型输入**和**完整可独立运行的awk脚本**，均已用 `awk` 实际跑过验证。

### 示例1 —— 字段锚点 + 固定字段偏移

**场景**：某交换机QoS队列丢包报表。`Interface:Queue` 形如 `eth0:0` 的复合ID是这一行里唯一可识别的锚点；同一接口有多个队列时，续行会把前导的 `Interface` 列省略为空，awk默认按空白分割时空字段直接消失，导致整行字段整体左移。

**典型输入**：

```
Interface  QueueID    Depth  Enqueued  Dropped
eth0       eth0:0     64     102394    12
           eth0:1     64     88210     0
           eth0:2     64     4021      340
eth1       eth1:0     32     55012     0
           eth1:1     32     901       5
```

**完整awk脚本**：

```awk
NR == 1 { next }
{
	for (i = 1; i <= NF; i++) {
		if ($i ~ /^[A-Za-z0-9]+:[0-9]+$/) {
			print $i, "dropped=" $(i + 3)
			break
		}
	}
}
```

**运行结果**：

```
eth0:0 dropped=12
eth0:1 dropped=0
eth0:2 dropped=340
eth1:0 dropped=0
eth1:1 dropped=5
```

**心智模式**：首行（`eth0 eth0:0 ...`）锚点落在 `$2`，`Dropped` 是 `$5`，即 `i+3`；续行（`eth0:1 ...`）锚点落在 `$1`，`Dropped` 变成 `$4`，仍然是 `i+3`。同一套“锚点+3”公式在两种列偏移下都能取对——**关键不是锚点在哪一列，而是锚点和目标字段之间隔着固定数量的列**。

### 示例2 —— 锚点即答案（自锚定，偏移量为0）

**场景**：格式不统一的DHCP/ARP日志，要提取IPv4地址，但IP在每行里出现的列位置都不一样。

**典型输入**：

```
Jul 16 09:12:01 dhcpd: lease 192.168.1.55 for client 3c:22:fb:aa:bb:cc
Jul 16 09:12:03 dhcpd: DHCPACK on 192.168.1.77 to 3c:22:fb:11:22:33 via eth0
Jul 16 09:12:05 arp-scan: 192.168.1.90 aa:bb:cc:dd:ee:ff reachable
```

**完整awk脚本**：

```awk
{
	for (i = 1; i <= NF; i++) {
		if ($i ~ /^([0-9]{1,3}\.){3}[0-9]{1,3}$/) {
			print $i
			break
		}
	}
}
```

**运行结果**：

```
192.168.1.55
192.168.1.77
192.168.1.90
```

**心智模式**：第一行IP在 `$6`，第二行在 `$7`，第三行在 `$5`——位置各不相同，但正则能在任意位置唯一识别出IP（MAC地址含 `:` 不会误命中）。这里**锚点本身就是目标值**，不需要额外偏移，是“锚点+偏移”思想里偏移量退化为0的特例：当目标本身就有独特模式时，找到即取，不用再算相对位置。

### 示例3 —— 行内锚点 + 动态区间扫描（偏移是区间，不是定长）

**场景**：温度传感器巡检日志。每行以站点号开头，`temps:` 后跟不定数量的读数，直到 `CHK=` 字段结束；不同站点的读数个数不一样，不能用固定偏移取值。

**典型输入**：

```
SITE-01 STATUS start temps: 21.3 22.0 23.5 22.8 CHK=ok
SITE-02 STATUS start temps: 19.9 20.1 CHK=ok
SITE-03 STATUS start temps: 30.2 31.0 29.8 33.4 34.1 CHK=warn
```

**完整awk脚本**（统计每个站点超过30.0的读数个数）：

```awk
{
	for (i = 1; i <= NF; i++) {
		if ($i == "temps:") {
			count = 0
			for (j = i + 1; j <= NF; j++) {
				if ($j ~ /^CHK/) break
				if ($j + 0 > 30.0) count++
			}
			print $1, count
			break
		}
	}
}
```

**运行结果**：

```
SITE-01 0
SITE-02 0
SITE-03 4
```

**心智模式**：`temps:` 是锚点，取值范围从 `i+1` 一直扫到遇到 `CHK` 前缀字段为止，而不是固定取 `i+1` 到 `i+4`。SITE-01/02/03 的读数个数分别是4/2/5个，脚本都能正确处理——**偏移可以是一个区间，用终止条件代替固定长度**。

### 示例4 —— 表头文本锚点，建立“列名→下标”映射

**场景**：两台主机/两个版本的进程报表，列的顺序不一致（甚至列数不同），固定 `$N` 会在不同格式间取错列；改为先解析表头文本，建立列名到下标的映射，按列名取值即可同时兼容两种格式。

**典型输入（hostA.txt）**：

```
PID   USER   %CPU  %MEM  COMMAND
1001  alice  12.3  4.5   nginx
1002  bob    0.1   0.2   sshd
```

**典型输入（hostB.txt，列序不同）**：

```
USER   PID   COMMAND   %MEM  %CPU
alice  1001  nginx     4.5   12.3
bob    1002  sshd      0.2   0.1
```

**完整awk脚本**（同一份脚本处理两种格式）：

```awk
FNR == 1 {
	delete col
	for (i = 1; i <= NF; i++) col[$i] = i
	next
}
{
	print $(col["USER"]), $(col["PID"]), $(col["%CPU"])
}
```

用 `awk -f script.awk hostA.txt hostB.txt` 运行，**运行结果**：

```
alice 1001 12.3
bob 1002 0.1
alice 1001 12.3
bob 1002 0.1
```

两份文件表头列序完全不同，但输出结果一致。**心智模式**：锚点从“某个字段的内容特征”变成了“表头行的列名文本”，`col[名字]` 就是“偏移”，本质仍是“先定位、再用相对关系取值”，只是把相对关系提前存进了一张表。

### 示例5 —— 行锚点 + 行方向偏移（多行才构成一条记录）

**场景**：类似 `ip -s link show` 的网卡收发统计，接口名和RX/TX统计项分别在独立的行上，真正的数值出现在标签行的**下一行**。

**典型输入**：

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc mq state UP
    link/ether 3c:22:fb:aa:bb:cc brd ff:ff:ff:ff:ff:ff
    RX: bytes  packets  errors  dropped overrun mcast
    582910233   431022   0       0       0       120
    TX: bytes  packets  errors  dropped carrier collsns
    390123001   210044   0       0       0       0
3: eth1: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN
    link/ether aa:bb:cc:dd:ee:ff brd ff:ff:ff:ff:ff:ff
    RX: bytes  packets  errors  dropped overrun mcast
    0           0        0       0       0       0
    TX: bytes  packets  errors  dropped carrier collsns
    0           0        0       0       0       0
```

**完整awk脚本**：

```awk
$1 ~ /^[0-9]+:$/ && $2 ~ /:$/ {
	iface = $2
	sub(/:$/, "", iface)
	next
}
$1 == "RX:" { want = "rx"; next }
$1 == "TX:" { want = "tx"; next }
want == "rx" { print iface, "RX bytes:", $1; want = ""; next }
want == "tx" { print iface, "TX bytes:", $1; want = "" }
```

**运行结果**：

```
eth0 RX bytes: 582910233
eth0 TX bytes: 390123001
eth1 RX bytes: 0
eth1 TX bytes: 0
```

**心智模式**：`RX:`/`TX:` 所在的行是锚点，但锚点行本身不含数值，真正要取的值在**下一行的 `$1`**——用一个状态变量 `want` 把“锚点已找到”这个信息带到下一条记录里，实现跨行的偏移。这说明偏移不一定发生在同一行的字段方向上，也可以发生在**行方向**上（下一行/下N行），且锚点行与取值行是分离的。

### 示例6 —— 本仓库真实案例：DOCSIS xmit-chan-counter，与示例1同构

前五个示例都是为了讲解思路而构造的典型输入。这一条换成本仓库里正在跑的真实代码，作为示例1（字段锚点+固定字段偏移）在生产环境里的实际印证。

**场景**：`show ccap docsis docs-mac-domain mac-domain modem <MAC> xmit-chan-counter | tab | nomore` 的输出。`UCID` 列格式为 `x/y`（`x`是子域号，`y`是channel号），是这一行里唯一的“数字/数字”格式字段；同一个CM的多个channel续行时，前导的 `MAC_DOMAIN_NAME`、`MODEM_MAC` 两列被省略为空，awk按空白分割后这两列直接消失，导致后面所有字段整体左移2位。目标是统计 channel 12 和 13 各自的 `UNCORRECTABLE` 计数。

**典型输入**（真实设备采集到的数据，见下图）：

```
MAC DOMAIN NAME     MODEM MAC          UCID  SID   BYTES        PACKETS   MIN MODULATION  MAX MODULATION  NO ERROR   CORRECTED  UNCORRECTABLE  NO BURST
astatine0_csl_md_0  38:ad:2b:5b:bb:98  0/13  1118  11855546661  13992717  qam1024         qam1024         16156137   102        15             70
                                       0/12  1118  84652        90        qam1024         qam1024         95         0          0              0
                                       0/0   1118  0            0         qam64           qam64           0          0          0              0
                                       0/1   1118  1069         1         qam64           qam64           5          0          0              0
                                       0/2   1118  0            0         qam64           qam64           0          0          0              0
                                       0/3   1118  0            0         qam64           qam64           0          0          0              0
```

**完整awk脚本**（原样摘自 [monitor_csl_ofdma_uncorrectable.sh:42-55](monitor_csl_ofdma_uncorrectable.sh)）：

```awk
{
	for (i = 1; i <= NF; i++) {
		if ($i ~ /^[0-9]+\/[0-9]+$/) {
			split($i, ucid, "/")
			chan = ucid[2] + 0
			if (chan == 12 || chan == 13) {
				sum[chan] += $(i + 8) + 0
			}
			break
		}
	}
}
END { printf "%d %d\n", sum[12]+0, sum[13]+0 }
```

**运行结果**：

```
0 15
```

即 channel 12 的 `UNCORRECTABLE` 合计为 `0`，channel 13 为 `15`——与截图里 `0/13` 那一行 `UNCORRECTABLE=15`、`0/12` 那一行 `UNCORRECTABLE=0` 完全一致。

**心智模式**：首行（`astatine0_csl_md_0 38:ad:2b:5b:bb:98 0/13 ...`）锚点 `0/13` 落在 `$3`，`UNCORRECTABLE` 是 `$11`，即 `i+8`；续行（`0/12 ...`）锚点落在 `$1`，`UNCORRECTABLE` 变成 `$9`，仍然是 `i+8`。这跟示例1的“锚点+3”是完全同一套心智模式，只是这里锚点右边隔着 `SID、BYTES、PACKETS、MIN_MOD、MAX_MOD、NO_ERROR、CORRECTED` 共7个字段，所以偏移量是 `i+8`（第8个才是目标）——**偏移量本身由具体协议/命令的列结构决定，思路不变，只需要照着真实表头数一遍列数即可确定**。

## 6. 心智模式小结

| 维度 | 示例1 | 示例2 | 示例3 | 示例4 | 示例5 | 示例6 |
| --- | --- | --- | --- | --- | --- | --- |
| 锚点类型 | 字段内容特征（复合ID） | 字段内容特征（IP格式） | 字段内容特征（关键字） | 表头文本（列名） | 行内容特征（RX:/TX:） | 字段内容特征（UCID） |
| 偏移类型 | 固定字段偏移 `i+k` | 偏移为0（自锚定） | 动态区间扫描直到终止符 | 按名字查表 | 行方向偏移（下一行） | 固定字段偏移 `i+k` |
| 典型适用场景 | 续行导致列整体错位 | 多格式混排，目标值位置不定 | 不定长的数值列表 | 多工具/多版本列序不一致 | 一条逻辑记录跨越多行 | 真实DOCSIS命令输出（生产案例） |

**一句话判断准则**：先看“目标值本身有没有独特模式”——如果有，直接用示例2的思路自锚定，偏移为0；如果没有，再看“锚点和目标值之间的相对关系”是定长（示例1、示例6）、区间（示例3）、按名字查表（示例4），还是跨行（示例5），据此选择对应的偏移手法。示例6证明这套心智模型不只是教学示例，在真实生产脚本里同样可靠。
